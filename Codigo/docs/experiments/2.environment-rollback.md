# Experiment: Environment Rollback

> Last documented update: 2026-03-19  
> Author: Mariana Almeida

## Objective

Demonstrate that a corrupted execution environment can be restored instantly using VM snapshots.

This experiment validates the concept of **environment rollback**, which is essential
for systems that perform automated experimentation such as AI agents.

## Background

During automated execution, environments may become corrupted due to:

- dependency conflicts
- file deletions
- configuration errors
- runtime crashes

Traditional recovery methods require rebuilding the environment or reinstalling dependencies.
Snapshots allow the system to revert instantly to a known good state.

## Requirements

Same setup as the previous experiment:

- Linux with KVM enabled
- Firecracker installed
- kernel image
- root filesystem

A working microVM environment must be available.

## Setup

Start a microVM using the previously configured environment.

Ensure that the VM can run commands inside the system.

## Steps

1. Start the microVM.

2. Launch a simple service inside the VM.

Example:

```bash
python3 -m http.server
```

3. Confirm the service is running.

4. Create a snapshot of the VM.

Example snapshot files:

```
snapshot.snap
memory.snap
```

5. Intentionally corrupt the environment.

Examples:

- delete application files
- stop running services
- remove system dependencies

6. Confirm that the system is broken.

7. Restore the VM snapshot.

## Expected Result

After restoring the snapshot:

- the service should be running again
- deleted files should be restored
- the environment should match the snapshot state

## Validation

The rollback mechanism is validated if:

- the system returns to its previous state
- no manual rebuild is required
- restoration occurs significantly faster than rebuilding the environment

## Notes

This experiment demonstrates one of the main advantages of snapshot-based environments:
**instant recovery from failure**.
