# Experiment: Running csv-analytics-benchmark inside Firecracker

> Last documented update: 2026-03-20  
> Author: Codex Agent

## Objective

Validate that the `csv-analytics-benchmark` workload can boot and finish inside a
Firecracker microVM using the custom rootfs built for SnapVM, and compare its
performance with the host execution.

## What was added

- `scripts/setup_firecracker_assets.sh`: end-to-end builder that
  1. reuses Docker to assemble a Ubuntu 22.04 rootfs with OpenJDK 21, BusyBox, and the benchmark files;
  2. emits `firecracker-assets/rootfs.ext4` + downloads `vmlinux.bin` and the
     `firecracker`/`jailer` binaries under `firecracker-bin/`.
- `scripts/run_firecracker_benchmark.sh`: starts Firecracker (API socket lives in `/tmp`
  to avoid WSL socket limitations), configures the VM, starts it, captures console output,
  and writes summary numbers to `firecracker-run/run_summary.txt`.

The benchmark rootfs uses a custom `/init` that prints both the Java-internal timer
(`JAVA_INTERNAL_MS`) and an additional `FIRECRACKER_WALL_MS` measured with `date +%s%3N`.
The init script ends with `busybox poweroff`, so Firecracker exits right after the workload.

## Reproduce the run

```bash
# 1) build assets (runs Docker + Maven)
./Codigo/scripts/setup_firecracker_assets.sh

# 2) execute the workload inside Firecracker
./Codigo/scripts/run_firecracker_benchmark.sh

# 3) optional: host baseline for comparison
cd Codigo/benchmark
/usr/bin/time -f 'HOST_REAL_MS=%E' java -Xms256m -Xmx768m -jar target/csv-analytics-benchmark.jar
```

Artifacts are stored under `Codigo/firecracker-assets/` (kernel + rootfs),
`Codigo/firecracker-bin/` (binaries), and `Codigo/firecracker-run/` (logs, summaries).
These directories are not tracked by git because the rootfs is ~2 GB.

## Results

| Environment | Wall-clock (host) | Guest wall (`FIRECRACKER_WALL_MS`) | Java internal (`JAVA_INTERNAL_MS`) |
|-------------|-------------------|------------------------------------|------------------------------------|
| Host JVM    | ~1.06 s (`/usr/bin/time`) | n/a | 850 ms |
| Firecracker | 8.49 s (VM lifecycle) | 2674 ms | 1614 ms |

Console + Firecracker logs: `Codigo/firecracker-run/firecracker-console.log`,
`firecracker.log`, `firecracker-metrics.log`.

## Observations

- The host JVM run (no isolation) finishes its internal work in ~850 ms and the full process
  takes ~1.06 s.
- Inside Firecracker the Java code itself reports ~1.6 s and the init script measures ~2.7 s,
  while the end-to-end VM lifecycle (start VM + boot + workload + shutdown) takes ~8.5 s.
  The delta captures Firecracker process startup, kernel boot, and shutdown.
- Rootfs boot required setting `is_root_device=false` in the drive config so we could
  override the `root=/dev/vda` boot argument (the disk image has no partition table).

## Next steps

- Capture multiple iterations to establish variance and warm-up effects.
- Extend the init script to emit more granular timings (e.g. splitting kernel boot,
  JVM startup, workload).
- Automate CSV ingestion from the console log to feed plotting/notebook tooling.
